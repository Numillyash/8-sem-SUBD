# Фрагменты для отчета по ЛР5

## Описание экспериментальной базы данных

Лабораторная работа выполнялась над базой данных, разработанной в лабораторной работе №1 для варианта №8. В базе данных использовались два отношения: `documents` и `document_operations`.

Таблица `documents` хранит сведения о документах организации: номер документа, тип документа, ответственного сотрудника, содержимое документа в формате `jsonb`, внутреннее подразделение, дату ввода в действие, дату окончания действия и степень секретности.

Таблица `document_operations` хранит журнал операций с документами: ФИО сотрудника, номер документа, дату изменения и описание операции с документом.

Для проведения эксперимента база данных была заполнена синтетическими тестовыми данными. В таблицу `documents` было добавлено 100000 записей, в таблицу `document_operations` — 300000 записей. После заполнения размер таблицы `documents` составил 33 MB, полный размер с индексами — 54 MB. Размер таблицы `document_operations` составил 34 MB, полный размер с индексами — 58 MB.

Для актуализации статистики оптимизатора после заполнения данных выполнялись команды `ANALYZE documents` и `ANALYZE document_operations`.

## Использованные запросы

### Запрос Q1

Запрос выбирает секретные действующие документы из нескольких подразделений. Запрос использует фильтрацию по подразделению, степени секретности и сроку действия документа, а также сортировку результата по дате окончания действия и номеру документа.

    SELECT
        document_number,
        document_type,
        responsible_employee,
        internal_department,
        effective_from,
        effective_until,
        secrecy_level
    FROM documents
    WHERE internal_department IN ('Подразделение_3', 'Подразделение_7', 'Подразделение_13')
      AND secrecy_level = 'секретно'
      AND effective_until IS NOT NULL
      AND effective_until >= DATE '2024-01-01'
    ORDER BY effective_until, document_number;

До оптимизации план выполнения запроса использовал последовательное сканирование таблицы `documents`. В плане присутствовала операция `Seq Scan on public.documents`. При выполнении запроса СУБД просматривала всю таблицу документов и отбрасывала большую часть строк по условию фильтрации. В исходном плане было отброшено 97940 строк, а возвращено 2060 строк. Это показывает, что без дополнительного индекса запрос имеет низкую эффективность: таблица читается полностью, хотя результат составляет небольшую часть от общего числа записей.

Для оптимизации был создан частичный составной индекс:

    CREATE INDEX idx_documents_q1_secret_department_until
    ON documents
    (
        secrecy_level,
        internal_department,
        effective_until,
        document_number
    )
    INCLUDE
    (
        document_type,
        responsible_employee
    )
    WHERE secrecy_level IS NOT NULL
      AND effective_until IS NOT NULL;

После создания индекса план изменился: вместо последовательного сканирования стал использоваться `Bitmap Index Scan` по индексу `idx_documents_q1_secret_department_until` и затем `Bitmap Heap Scan` по таблице `documents`. Это означает, что PostgreSQL сначала находит подходящие строки по индексу, а затем обращается к соответствующим страницам таблицы.

Среднее время выполнения Q1 уменьшилось с 11.494 ms до 2.335 ms. Коэффициент ускорения составил 4.923 раза, а снижение времени выполнения — 79.69%.

### Запрос Q2

Запрос выполняет соединение журнала операций с таблицей документов и считает количество операций сотрудника за заданный период по ответственным сотрудникам и подразделениям.

    SELECT
        d.responsible_employee,
        d.internal_department,
        count(*) AS operations_count
    FROM document_operations AS o
    JOIN documents AS d
        ON d.document_number = o.document_number
    WHERE o.employee_name = 'Сотрудник_В'
      AND o.change_date >= TIMESTAMP '2024-06-01 00:00:00'
      AND o.change_date <  TIMESTAMP '2024-09-01 00:00:00'
      AND d.secrecy_level IS NOT NULL
    GROUP BY
        d.responsible_employee,
        d.internal_department
    ORDER BY
        operations_count DESC,
        d.responsible_employee,
        d.internal_department;

До оптимизации план выполнения использовал параллельное последовательное сканирование таблиц `document_operations` и `documents`, после чего выполнялось соединение `Parallel Hash Join`, группировка и сортировка. Основным узким местом было чтение значительного количества строк из таблицы `document_operations` с последующим отбрасыванием строк по условиям `employee_name` и `change_date`.

Для оптимизации были созданы индексы:

    CREATE INDEX idx_operations_q2_employee_date_document
    ON document_operations
    (
        employee_name,
        change_date,
        document_number
    );

    CREATE INDEX idx_documents_q2_secret_document_employee_department
    ON documents
    (
        document_number,
        responsible_employee,
        internal_department
    )
    WHERE secrecy_level IS NOT NULL;

После оптимизации PostgreSQL начал использовать `Index Only Scan` по таблице `document_operations` и `Index Only Scan` по таблице `documents`. Это позволило уменьшить объем чтения таблиц и сократить время выполнения запроса. При этом в плане осталась операция `Hash Join`, так как запрос все равно требует соединить строки двух отношений и затем выполнить группировку.

Среднее время выполнения Q2 уменьшилось с 21.446 ms до 10.519 ms. Коэффициент ускорения составил 2.039 раза, снижение времени выполнения — 50.95%.

### Запрос Q3

Запрос определяет количество несекретных документов по подразделениям, для которых в журнале операций есть операция изменения срока действия за заданный период. Запрос использует условие `EXISTS`, то есть проверяет наличие связанных записей в таблице `document_operations`.

    SELECT
        d.internal_department,
        count(*) AS changed_documents
    FROM documents AS d
    WHERE d.secrecy_level IS NULL
      AND EXISTS
      (
          SELECT 1
          FROM document_operations AS o
          WHERE o.document_number = d.document_number
            AND o.operation_description = 'Изменение срока действия'
            AND o.change_date >= TIMESTAMP '2024-01-01 00:00:00'
            AND o.change_date <  TIMESTAMP '2025-01-01 00:00:00'
      )
    GROUP BY
        d.internal_department
    ORDER BY
        changed_documents DESC,
        d.internal_department;

До оптимизации PostgreSQL использовал `Parallel Seq Scan` по таблице `documents` и `Parallel Seq Scan` по таблице `document_operations`, затем выполнял `Parallel Hash Semi Join`. Последовательное сканирование таблицы операций было затратным, так как фильтр по `operation_description` и диапазону `change_date` применялся уже после чтения большого числа строк.

Для оптимизации были созданы индексы:

    CREATE INDEX idx_operations_q3_operation_date_document
    ON document_operations
    (
        operation_description,
        change_date,
        document_number
    );

    CREATE INDEX idx_documents_q3_public_document_department
    ON documents
    (
        document_number,
        internal_department
    )
    WHERE secrecy_level IS NULL;

После оптимизации план стал использовать `Index Only Scan` по индексу `idx_operations_q3_operation_date_document` и `Index Only Scan` по индексу `idx_documents_q3_public_document_department`. Операция соединения была представлена как `Hash Semi Join`, что соответствует семантике условия `EXISTS`: для каждой строки из `documents` требуется проверить наличие хотя бы одной подходящей строки в `document_operations`.

Среднее время выполнения Q3 уменьшилось с 40.708 ms до 19.614 ms. Коэффициент ускорения составил 2.075 раза, снижение времени выполнения — 51.82%.

## Сводные результаты измерений

Измерение времени выполнялось по 5 запускам для каждого запроса до и после оптимизации. Для уменьшения влияния случайных факторов перед серией измерений выполнялся один прогревочный запуск.

Сводные результаты:

- Q1: среднее время уменьшилось с 11.494 ms до 2.335 ms, ускорение составило 4.923 раза.
- Q2: среднее время уменьшилось с 21.446 ms до 10.519 ms, ускорение составило 2.039 раза.
- Q3: среднее время уменьшилось с 40.708 ms до 19.614 ms, ускорение составило 2.075 раза.

Наиболее значительное ускорение получено для Q1, так как запрос имеет сравнительно высокую селективность и использует условия фильтрации по нескольким индексируемым атрибутам. После создания индекса СУБД перестала выполнять полное последовательное чтение таблицы `documents`.

Для Q2 и Q3 ускорение оказалось умеренным. Это объясняется тем, что данные запросы не только фильтруют строки, но и выполняют соединение, группировку и сортировку. Даже после применения индексов часть стоимости выполнения остается связанной с обработкой промежуточного результата.

## Вывод по оптимизации

В ходе работы были исследованы планы выполнения трех запросов к базе данных документов организации. На исходном этапе во всех запросах были обнаружены операции последовательного сканирования таблиц, что приводило к чтению большого количества строк и последующему отбрасыванию значительной части данных по условиям фильтрации.

Для оптимизации были созданы составные и частичные индексы, соответствующие условиям фильтрации и соединения в запросах. После добавления индексов PostgreSQL начал использовать `Bitmap Index Scan`, `Bitmap Heap Scan` и `Index Only Scan`. Это позволило сократить объем чтения таблиц и снизить среднее время выполнения всех исследованных запросов.

Полученные результаты подтверждают, что эффективность индекса зависит от селективности условий и структуры запроса. Наибольший эффект достигается в тех случаях, когда индекс позволяет быстро отобрать небольшую часть строк из большой таблицы. В запросах с соединениями и агрегацией индексы также дают прирост производительности, однако итоговое время выполнения дополнительно зависит от стоимости соединения, группировки и сортировки.
