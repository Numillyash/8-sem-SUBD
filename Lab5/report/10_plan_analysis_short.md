# Краткий анализ планов выполнения запросов

## Запрос Q1

### Исходный план

В исходном плане запроса Q1 использовалось последовательное сканирование таблицы `documents`:

    Seq Scan on public.documents
    Rows Removed by Filter: 97940
    Execution Time: 26.670 ms

Основная проблема исходного плана заключалась в том, что PostgreSQL просматривал всю таблицу `documents`, содержащую 100000 записей, и только после этого применял условия фильтрации по подразделению, степени секретности и сроку действия документа. При этом большая часть строк отбрасывалась фильтром.

### План после оптимизации

После создания индекса `idx_documents_q1_secret_department_until` план изменился:

    Bitmap Index Scan on idx_documents_q1_secret_department_until
    Bitmap Heap Scan on public.documents
    Execution Time: 8.878 ms

PostgreSQL стал использовать индекс для предварительного отбора строк, соответствующих условиям `secrecy_level`, `internal_department` и `effective_until`. За счет этого уменьшился объем данных, читаемых из таблицы.

### Результат

По серии из пяти измерений среднее время выполнения Q1 уменьшилось с 11.494 ms до 2.335 ms. Ускорение составило 4.923 раза, снижение времени выполнения — 79.69%.

## Запрос Q2

### Исходный план

В исходном плане запроса Q2 использовались параллельные последовательные сканирования таблиц `document_operations` и `documents`:

    Parallel Seq Scan on public.document_operations
    Parallel Seq Scan on public.documents
    Parallel Hash Join
    Execution Time: 34.391 ms

Основная проблема заключалась в чтении большого объема строк из `document_operations` с последующей фильтрацией по сотруднику и диапазону дат. Также выполнялось чтение таблицы `documents` для отбора документов со степенью секретности.

### План после оптимизации

После создания индексов `idx_operations_q2_employee_date_document` и `idx_documents_q2_secret_document_employee_department` план изменился:

    Index Only Scan using idx_operations_q2_employee_date_document
    Index Only Scan using idx_documents_q2_secret_document_employee_department
    Hash Join
    Execution Time: 28.945 ms

PostgreSQL начал использовать индексное чтение без обращения к строкам таблицы для части данных. Операция соединения осталась в плане, так как запрос требует объединения записей из двух таблиц и последующей группировки.

### Результат

По серии из пяти измерений среднее время выполнения Q2 уменьшилось с 21.446 ms до 10.519 ms. Ускорение составило 2.039 раза, снижение времени выполнения — 50.95%.

## Запрос Q3

### Исходный план

В исходном плане запроса Q3 использовались параллельные последовательные сканирования и полусоединение:

    Parallel Seq Scan on public.documents
    Parallel Seq Scan on public.document_operations
    Parallel Hash Semi Join
    Execution Time: 49.575 ms

Узким местом являлось последовательное сканирование таблицы `document_operations`, где фильтр по описанию операции и диапазону дат применялся после чтения большого количества строк. Также таблица `documents` сканировалась для отбора несекретных документов.

### План после оптимизации

После создания индексов `idx_operations_q3_operation_date_document` и `idx_documents_q3_public_document_department` план изменился:

    Index Only Scan using idx_operations_q3_operation_date_document
    Index Only Scan using idx_documents_q3_public_document_department
    Hash Semi Join
    Execution Time: 39.098 ms

PostgreSQL стал использовать индекс для поиска операций с описанием `Изменение срока действия` в заданном диапазоне дат, а также частичный индекс по несекретным документам. Операция `Hash Semi Join` соответствует условию `EXISTS`, так как для документа проверяется наличие хотя бы одной подходящей операции.

### Результат

По серии из пяти измерений среднее время выполнения Q3 уменьшилось с 40.708 ms до 19.614 ms. Ускорение составило 2.075 раза, снижение времени выполнения — 51.82%.

## Общий вывод по планам

До оптимизации во всех запросах использовались операции последовательного сканирования таблиц. Это приводило к чтению большого количества строк и последующему отбрасыванию значительной части данных фильтрами.

После добавления индексов PostgreSQL начал использовать `Bitmap Index Scan`, `Bitmap Heap Scan` и `Index Only Scan`. Наибольший эффект был получен для Q1, так как условия запроса обладают высокой селективностью и хорошо соответствуют созданному составному индексу.

Для Q2 и Q3 ускорение также было получено, однако оно оказалось меньше, чем для Q1. Причина заключается в том, что эти запросы дополнительно выполняют соединение таблиц, группировку и сортировку, поэтому итоговое время зависит не только от скорости поиска строк, но и от стоимости обработки промежуточного результата.
